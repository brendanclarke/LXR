# STM32F4 Setup & Resource Allocation — LXR Audio Mainboard

## 1. Overview

This document details the timing, interrupt, clock tree, DMA, and memory resource allocation in the STM32F407VGT6 audio mainboard firmware. The processor runs at **168 MHz** sourced from an 8 MHz HSE crystal through the main PLL. The PLLI2S provides a separate clock chain for accurate I2S audio timing.

---

## 2. Clock Tree Configuration

Configured in `system_stm32f4xx.c` (generated by ST's clock configuration tool):

```
HSE (8 MHz)
  └─ PLL_M = 8 → VCO in = 1 MHz
       └─ PLL_N = 336 → VCO out = 336 MHz
            ├─ PLL_P = 2 → SYSCLK = 168 MHz
            └─ PLL_Q = 7 → 48 MHz (USB OTG FS, RNG, SDIO)

SYSCLK (168 MHz)
  └─ AHB prescaler = 1 → HCLK = 168 MHz
       ├─ APB1 prescaler = 4 → PCLK1 = 42 MHz
       │     └─ TIM2,3,4,5,6,7,12,13,14 clock = 84 MHz (×2 when APB≠1)
       │           ├─ USART2 (MIDI DIN, 31250 baud)
       │           ├─ USART3 (front panel, 500 kbaud)
       │           ├─ SPI2/I2S2 (DAC2 or ADC)
       │           ├─ SPI3/I2S3 (DAC1)
       │           └─ TIM2 (external clock measurement)
       └─ APB2 prescaler = 2 → PCLK2 = 84 MHz
             └─ TIM1,8,9,10,11 clock = 168 MHz
                   └─ SYSCFG (EXTI routing)

HSE (8 MHz)
  └─ PLLI2S_N = 383 → PLLI2S VCO = 383 MHz
       └─ PLLI2S_R = 2 → I2S kernel clock = 191.5 MHz
            └─ I2S prescaler (ODD=1, DIV=2): 191.5 MHz / (2×2+1) → ~44002.76 Hz I2S LRCK
```

**Flash wait states:** 5 WS (required at 168 MHz, 3.3 V). Instruction cache ON, data cache ON, prefetch OFF (as configured by ST tool).

## 3. Timer Allocation

The STM32F407 has 14 timers. Two are used by the application.

### 3.1 SysTick — 250 μs System Tick

| Parameter | Value |
|---|---|
| Source | HCLK (AHB clock, 168 MHz) |
| Reload value | `168,000,000 / 4000 = 42,000` |
| Tick period | **250 μs (0.25 ms)** |
| Counter width | 24-bit |
| ISR | `SysTick_Handler` |

`SysTick_Handler` increments `volatile uint32_t systick_ticks` (stored in CCM RAM for fast access). This counter drives:

- **Sequencer tempo:** `seq_tick()` fires `seq_nextStep()` when `systick_ticks - seq_lastTick >= seq_deltaT`. At 120 BPM a sequencer step (at 32 ppq / 4 main steps per beat) occurs every `(1000 × 60) / 120 / 96 × 4 = 20.83 ms` = **83.3 SysTick ticks**.
- **MIDI clock sync:** `clockSync.c` uses `systick_ticks` to measure the interval between received MIDI clock bytes (24 ppq) and compute a rolling BPM average.
- **Trigger pulse duration:** `PULSE_LENGTH = PULSE_LENGTH_MS × 4 = 20 × 4 = 80 ticks = 20 ms`.
- **USB tick management** via `usb_tick()` in the main loop.

**Temporal resolution:** 250 μs per tick. At 120 BPM the sequencer error introduced by SysTick granularity is at most ±250 μs per step. The code corrects for this by subtracting the overshoot (`rest = systick_ticks - seq_lastTick - seq_deltaT`) from the next `deltaT`.

### 3.2 TIM2 — External Clock Pulse Measurement

| Parameter | Value |
|---|---|
| Peripheral | TIM2 (32-bit general-purpose, APB1 domain) |
| APB1 timer clock | 84 MHz |
| Prescaler register (TIM_Prescaler field) | 1 |
| Effective clock | 84 MHz / (1+1) = **42 MHz** |
| Resolution | 1/42,000,000 s ≈ **23.8 ns per count** |
| Period | 0xFFFFFFFF (maximum, ~102 seconds before overflow) |
| Mode | Free-running upcounting |
| Interrupt | None (polled in EXTI ISR) |

`COUNTER_DURATION = 1.0f / 42,000,000` converts timer counts to seconds. On each clock input edge (`EXTI_Line9` falling), the ISR reads the counter, passes it through a median filter (three-sample), computes `f = 1 / (count × COUNTER_DURATION / prescaler)`, derives `trigger_phaseInc`, and resets the counter to zero for the next measurement. This allows the phase accumulator to interpolate between external clock pulses for smooth per-sample sequencer advancement.

### 3.3 Timers Not Used by Application

TIM1, TIM3–TIM14 are not configured by the application firmware. They are available for future use.

## 4. Interrupt Allocation

### 4.1 Interrupt Priority Groups

The Cortex-M4 NVIC uses 4 priority bits. The firmware appears to use the default priority grouping (4 bits preemption, 0 sub-priority bits based on the EVAL_AUDIO defines). Observed priority assignments:

| IRQ | Preemption Priority | Sub-Priority | Description |
|---|---|---|---|
| `DMA1_Stream7_IRQn` | 0 (highest) | 0 | I2S DAC1 DMA transfer complete |
| `DMA1_Stream4_IRQn` | 0 | 0 | I2S DAC2 DMA transfer complete |
| `DMA1_Stream3_IRQn` | 0 | 0 | I2S ADC DMA transfer complete |
| `OTG_FS_IRQHandler` | (USB library default) | — | USB OTG FS |
| `USART2_IRQHandler` | (NVIC default) | — | MIDI DIN UART |
| `USART3_IRQHandler` | (NVIC default) | — | Front panel UART |
| `EXTI9_5_IRQn` (Line 9, clock in) | 0x00 | 0x0F | External clock input — high priority |
| `EXTI9_5_IRQn` (Line 8, reset in) | 0x0F | 0x0F | External reset input — lowest priority |
| `SysTick` | (Cortex default) | — | 250 μs system tick |

**Shared IRQ channel:** EXTI lines 5–9 all share `EXTI9_5_IRQn`. The handler checks `EXTI_GetITStatus()` for each of Line 8 and Line 9 separately.

### 4.2 Audio DMA ISR Detail (`DMA1_Stream7_IRQHandler`)

This is the most timing-critical interrupt in the system. Fired every `OUTPUT_DMA_SIZE / REAL_FS = 32 / 44002.76 ≈ 726.9 μs`:

1. Checks `DMA_FLAG_TCIF7` (transfer complete).
2. Disables and de-initialises the stream.
3. Clears the flag.
4. Reconfigures `DMA_Memory0BaseAddr` to point to the alternate half of `dma_buffer` (ping-pong).
5. Sets `DMA_BufferSize = OUTPUT_DMA_SIZE * 2`.
6. Re-initialises and re-enables DMA1_Stream7.
7. Sets `bCurrentSampleValid = 1 - (dmaPtr & 1)` to signal the main loop that a new DSP block is needed.

The half-transfer interrupt (`DMA_FLAG_HTIF7`) is enabled but takes no action (placeholder for future use).

**Budget:** The ISR must complete well within 726.9 μs (the DMA period). With Cortex-M4 at 168 MHz this is 122,020 cycles — more than adequate for the ISR body, which is a handful of peripheral register writes.

## 5. DMA Allocation

| Stream | Channel | Peripheral | Direction | Priority | Use |
|---|---|---|---|---|---|
| DMA1 Stream7 | Ch 0 | SPI3 (I2S3) DR | Mem→Periph | High | DAC1 audio output |
| DMA1 Stream4 | Ch 0 | SPI2 (I2S2) DR | Mem→Periph | High | DAC2 audio output (if USE_DAC2=1) |
| DMA1 Stream3 | Ch 0 | SPI2 (I2S2) DR | Periph→Mem | High | ADC audio input (if USE_DAC2=0) |

All streams use:
- Half-word (16-bit) peripheral and memory data size.
- Memory increment enabled, peripheral increment disabled.
- Normal (non-circular) mode with manual re-arming in the TC ISR.
- FIFO mode disabled (direct mode).
- Single burst for both memory and peripheral.

**Why non-circular?** Circular mode would autonomously wrap the DMA address. Non-circular with manual re-arm gives the ISR explicit control to switch the buffer pointer between the two halves of the ping-pong buffer. The re-arm happens inside the TC ISR so the audio stream is continuous in practice, at the cost of a small window where the DMA is momentarily idle between `DMA_Cmd(DISABLE)` and `DMA_Cmd(ENABLE)` in the ISR.

## 6. Audio Timing Budget

| Metric | Value |
|---|---|
| Sample rate | 44002.76 Hz |
| DMA block size | 32 stereo samples = 64 int16 values |
| DMA block period | 32 / 44002.76 = **726.9 μs** |
| SysTick period | 250 μs |
| SysTick ticks per DMA block | ~2.9 |
| DSP time budget per block | < 726.9 μs (must complete before next DMA TC) |

The main loop calls `calcNextSampleBlock()` only when `bCurrentSampleValid != SAMPLE_VALID`. This flag is set by the DMA ISR. The DSP calculation (7 voices + mixer) must therefore complete within one DMA period. At 168 MHz with the hardware FPU active and flush-to-zero mode enabled (`FPU->FPDSCR |= FPU_FPDSCR_FZ_Msk`), the synthesis is expected to complete well within budget.

**Flush-to-zero (FTZ):** Enabled at startup to eliminate the CPU penalty of handling IEEE 754 denormal numbers, which can arise in decaying audio envelopes approaching zero.

## 7. Memory Allocation

### 7.1 Flash Layout

| Region | Address | Size | Contents |
|---|---|---|---|
| Bootloader sector | `0x08000000` | 16 KB (sector 0) | SD firmware update bootloader |
| Application vector table | `0x08004000` | — | ISR vectors, startup |
| Application code | `0x08004000+` | Up to 1008 KB | Firmware code, rodata |

The linker script offsets the Flash origin to `0x08004000` when `USE_BOOTLOADER=1`. The startup file (`startup_stm32f4xx.S`) and vector table must be placed at this address for the bootloader to correctly hand off execution.

### 7.2 Main SRAM (112 KB at 0x20000000)

| Region | Contents |
|---|---|
| `.data` | Initialised global variables (copied from Flash at startup) |
| `.bss` | Zero-initialised globals |
| `dma_buffer[128]` | Audio DMA buffer, DAC1 (256 bytes) |
| `dma_buffer2[128]` | Audio DMA buffer, DAC2/ADC (256 bytes) |
| FatFs / SD structures | File system state, 512-byte sector buffer |
| UART FIFOs | `Fifo` × 2 (MIDI TX/RX), `Fifo` + `FifoBig` (front TX/RX) |
| DSP state | Voice structs (DrumVoice × 3, Snare, HiHat, Cymbal), oscillators, filters, envelopes |
| Sequencer state | Pattern sets, step data, timing variables |
| Heap | Minimal (`_Min_Heap_Size = 0` in linker script — heap effectively not used) |

### 7.3 CCM RAM (64 KB at 0x10000000)

CCM (Core Coupled Memory) provides zero-wait-state access from the core (but cannot be used by DMA). It is used for the stack and time-critical variables:

| Variable | Section | Description |
|---|---|---|
| `systick_ticks` | `.ccm` (INCCM) | Global tick counter, incremented in SysTick ISR — must be fast |
| Application stack | CCM (top = `_ccm_stack = 0x10010000`) | Zero wait-state stack for all ISRs and function calls |

The CCM stack pointer is set in `startup_stm32f4xx.S`. Placing the stack in CCM gives a meaningful latency advantage for ISRs, which push/pop the stack heavily.

**DMA restriction:** CCM RAM is not accessible by DMA. Audio DMA buffers (`dma_buffer`, `dma_buffer2`) must therefore reside in main SRAM, not CCM.

### 7.4 Stack

| Parameter | Value |
|---|---|
| Location | CCM RAM |
| Top address | `0x10010000` (end of 64 KB CCM) |
| Minimum reserved | 0x400 = 1 KB |
| Available | Up to ~64 KB minus CCM data usage |

With minimal CCM data (primarily `systick_ticks`), approximately **63 KB** of CCM is available as stack, which is very generous for a bare-metal application.

## 8. Sequencer Timing Model

The sequencer uses a software timing model based on `systick_ticks` (250 μs resolution):

```
At 120 BPM, 4/4 time:
  - One quarter note = 500 ms = 2000 ticks
  - Internal resolution: 96 ppq
  - seq_deltaT = (1000 × 60 / BPM) / 96 × 4 ms
              = 500 / 96 × 4 = 20.83 ms = 83.3 ticks

  - One 32nd-note step = seq_deltaT × (96/32) / 4 ... 
    (the factor of 4 in the code compensates for the 96ppq internal clock)
```

`seq_tick()` in the main loop compares `systick_ticks - seq_lastTick` against `seq_deltaT` on every iteration. When the threshold is reached it calls `seq_nextStep()` and carries the overshoot forward into the next `deltaT` calculation to prevent tempo drift.

**Shuffle:** Applied by modifying `seq_deltaT` for alternating steps using a pre-computed lookup table (`seq_shuffleTable`). Even steps are shortened and odd steps are lengthened by `shuffleFactor × originalDeltaT × 16`.

**External sync:** When `seq_getExtSync()` is active, `seq_calcDeltaT()` is still called internally (to advance non-master steps), but master steps are only advanced when `seq_triggerNextMasterStep()` is called from `EXTI9_5_IRQHandler`. This creates a hybrid model: interpolated steps between real clock pulses and hard-synced steps on the pulse itself.

## 9. Startup Sequence

Executed from `main()` before the main loop:

1. `initSpiPins()` — Tri-states STM32 SPI pins to avoid contention with AVR-owned SD card.
2. `RCC_GetClocksFreq(&RCC_Clocks)` — Reads actual clock frequencies.
3. `SysTick_Config(HCLK / 4000)` — Configures SysTick for 250 μs period.
4. `initAudioJackDiscoverPins()` — Configures PA0, PA5, PC4, PC5 as inputs with pull-ups.
5. `mixer_init()` — Initialises audio mixer state; zeroes output buffer.
6. `calcNextSampleBlock()` — Pre-fills the first DMA buffer before the codec starts.
7. `CodecInit(SYNTH_FS)` — Configures I2S GPIO, initialises SPI3 and SPI2 in I2S mode at 44.1 kHz, sets up DMA1 Stream7 and Stream4/Stream3, starts DMA transfer. **Audio output begins here.**
8. `parameterArray_init()` — Initialises the MIDI CC → parameter mapping.
9. `trigger_init()` — Configures trigger/clock GPIO outputs and inputs, EXTI lines 8 and 9, and TIM2.
10. `modNode_init()` × 6 — Initialises velocity modulation nodes for each voice.
11. `initMidiUart()` — Configures USART2 at 31250 baud, enables NVIC.
12. `initFrontpanelUart()` — Configures USART3 at 500 kbaud, enables NVIC.
13. `initRng()` — Initialises hardware RNG (used for noise and random pattern generation).
14. `euklid_init()` — Initialises Euclidean rhythm generator state.
15. `som_init()` — Initialises Self-Organising Map pattern generator state.
16. `seq_init()` + `seq_setBpm(120)` — Initialises sequencer state at 120 BPM.
17. Zero-initialise MIDI channel and note override arrays.
18. `initDrumVoice()`, `Snare_init()`, `HiHat_init()`, `Cymbal_init()` — Initialise all 7 voice DSP structures.
19. `usb_init()` — Initialises USB OTG FS as a MIDI device.
20. `FPU->FPDSCR |= FPU_FPDSCR_FZ_Msk` — Enable flush-to-zero FPU mode.
21. Enter **main loop**.

## 10. Main Loop Structure

The main loop is a cooperative superloop with no RTOS. Each iteration:

| Step | Function | Notes |
|---|---|---|
| 1 | `usb_tick()` | Processes USB stack events |
| 2 | `uart_processMidi()` | Drains MIDI DIN FIFO, calls `midiParser_parseMidiMessage()` |
| 3 | `uart_processFront()` | Drains front-panel FIFO, processes AVR parameter/button messages |
| 4 | USB MIDI dequeue | `usb_getMidi(&msg)` → `midiParser_parseMidiMessage(msg)` if available |
| 5 | `calcNextSampleBlock()` | Runs **only if** `bCurrentSampleValid != SAMPLE_VALID` (set by DMA ISR) |
| 6 | `seq_tick()` | Advances sequencer if `seq_deltaT` elapsed; sends MIDI clock |
| 7 | `trigger_tick()` | Deactivates trigger outputs whose pulse duration has expired |

The DSP step (step 5) is the most computationally expensive, consuming the majority of the ~727 μs budget. Steps 1–4 and 6–7 are expected to complete in well under 100 μs combined.

**No sleep:** The processor never enters a sleep mode in the main loop. It runs at full 168 MHz continuously.
