# STM32F4 Hardware Reference — STM32F407VGT6 (LXR Audio Mainboard)

## 1. Overview

The LXR audio mainboard is built around an **STM32F407VGT6** Cortex-M4F running at **168 MHz**. It is responsible for all digital audio synthesis (seven drum voices, DSP effects, mixer), MIDI parsing (DIN and USB), the step sequencer, trigger/clock I/O, SD-card sample management, and communication with the ATmega644 front panel. The mainboard connects to the control board through two 2×13 female headers and physically piggy-backs underneath it.

Firmware entry point: `mainboard/LxrStm32/src/main.c`. Toolchain target: ARM Cortex-M4 with FPU, compiled with GCC.

## 3. Clock Tree

The clock configuration is generated by ST's clock configuration spreadsheet and frozen in `system_stm32f4xx.c`. All values are exact unless noted.

| Domain | Source | Frequency |
|---|---|---|
| HSE crystal | External | 8 MHz |
| PLL_M | ÷8 → VCO input | 1 MHz |
| PLL_N | ×336 → VCO output | 336 MHz |
| PLL_P (SYSCLK) | ÷2 | **168 MHz** |
| PLL_Q (USB/RNG/SDIO) | ÷7 | **48 MHz** (exact for USB) |
| AHB (HCLK) | SYSCLK ÷1 | 168 MHz |
| APB1 | HCLK ÷4 | **42 MHz** |
| APB2 | HCLK ÷2 | **84 MHz** |
| TIM on APB1 | APB1 × 2 | **84 MHz** |
| TIM on APB2 | APB2 × 2 | **168 MHz** |
| PLLI2S_N | 383 | — |
| PLLI2S_R | ÷2 | 191.5 MHz I2S PLL VCO |
| I2S clock | From PLLI2S | 191.5 MHz → prescaler ODD=1, DIV=2 → ~44.002 kHz actual sample rate |

The actual audio sample rate, accounting for prescaler rounding, is **44002.76 Hz** (`REAL_FS` constant in `config.h`). Code that requires exact frequency calculations (e.g. LFO sync, phase increment tables) uses this value rather than the nominal 44100 Hz.

## 4. Memory Map

| Region | Start | Size | Usage |
|---|---|---|---|
| Flash (application) | `0x08004000` | 1008 KB | Application code, rodata, initial values |
| Flash (bootloader sector) | `0x08000000` | 16 KB | SD-card firmware update bootloader (reserved) |
| Main SRAM | `0x20000000` | 112 KB | `.data`, `.bss`, heap, main stack |
| CCM RAM | `0x10000000` | 64 KB | Time-critical variables, CCM stack |

**CCM RAM usage:** Variables declared with `INCCM` (`__attribute__((section(".ccm")))`) are pre-loaded from Flash into CCM at startup by the startup file. Variables declared `INCCMZ` are zero-initialised in CCM at startup. The stack itself lives in CCM RAM (top = `0x10010000`, defined as `_ccm_stack` in the linker script), providing zero-wait-state stack access for ISRs.

Examples of CCM-allocated data from the source: `systick_ticks` (volatile global tick counter, placed in CCM for fast ISR access).

**Stack size:** Minimum stack reserved = 0x400 (1 KB), but the actual usable stack depends on runtime usage.

**DMA buffers:** The audio DMA buffers `dma_buffer[OUTPUT_DMA_SIZE*4]` and `dma_buffer2[OUTPUT_DMA_SIZE*4]` are `int16_t` arrays located in main SRAM. With `OUTPUT_DMA_SIZE=32`, each array is 32×4 = 128 `int16_t` values = **256 bytes**. The factor of ×4 comes from ×2 for stereo and ×2 for double-buffering (ping-pong). See Section 6.1 for DMA details.

## 5. GPIO Pin Allocation

### 5.1 Audio I2S — DAC1 (CS4344, primary stereo output)

Uses **SPI3 in I2S mode** on GPIOA and GPIOC:

| Signal | GPIO | Pin | AF |
|---|---|---|---|
| WS (word select / LRCK) | GPIOA | PA15 | `GPIO_AF_SPI3` |
| CK (bit clock) | GPIOC | PC10 | `GPIO_AF_SPI3` |
| SD (serial data) | GPIOC | PC12 | `GPIO_AF_SPI3` |
| MCK (master clock) | GPIOC | PC7  | `GPIO_AF_SPI3` |

### 5.2 Audio I2S — DAC2 / ADC (CS4344 second output or CS5343 input)

Uses **SPI2 in I2S mode** on GPIOB and GPIOC. Whether this is a second DAC or the ADC input is controlled at compile time by `USE_DAC2` in `config.h` (default: `USE_DAC2 = 1`, second DAC active):

| Signal | GPIO | Pin | AF |
|---|---|---|---|
| WS | GPIOB | PB12 | `GPIO_AF_SPI2` |
| CK | GPIOB | PB13 | `GPIO_AF_SPI2` |
| SD | GPIOB | PB15 | `GPIO_AF_SPI2` |
| MCK | GPIOC | PC6  | `GPIO_AF_SPI2` |

### 5.3 MIDI DIN UART — USART2

| Signal | GPIO | Pin | Direction |
|---|---|---|---|
| TX (MIDI OUT) | GPIOA | PA2 | Output (open-drain, pull-up) |
| RX (MIDI IN) | GPIOA | PA3 | Input (pull-down) |

### 5.4 Front Panel UART — USART3

| Signal | GPIO | Pin | Direction |
|---|---|---|---|
| TX → AVR RX | GPIOB | PB10 | Output (push-pull) |
| RX ← AVR TX | GPIOB | PB11 | Input (pull-down) |

### 5.5 Trigger / Gate Outputs

Seven per-voice trigger outputs, two clock outputs, and one reset output are driven directly from GPIO:

| Signal | GPIO | Pin | Note |
|---|---|---|---|
| Track 1 trigger | GPIOD | PD8  | Inverted: 0=active |
| Track 2 trigger | GPIOD | PD9  | |
| Track 3 trigger | GPIOD | PD10 | |
| Track 4 trigger | GPIOD | PD11 | |
| Track 5 trigger | GPIOD | PD12 | |
| Track 6 trigger | GPIOD | PD13 | |
| Track 7 trigger | GPIOD | PD14 | |
| Clock OUT 1 | GPIOD | PD15 | |
| Clock OUT 2 | GPIOA | PA9  | |
| Reset OUT | GPIOA | PA10 | |

All trigger outputs are push-pull, 50 MHz slew rate. Output active-low logic: the firmware calls `GPIOD->ODR &= ~pin` to assert and `GPIOD->ODR |= pin` to deassert, because the external trigger circuit inverts the signal.

### 5.6 Trigger / Clock / Reset Inputs (External Sync)

| Signal | GPIO | Pin | EXTI Line | Trigger |
|---|---|---|---|---|
| Reset IN | GPIOA | PA8 | EXTI Line 8 | Rising and falling |
| Clock IN | GPIOC | PC9 | EXTI Line 9 | Falling (inverted by input transistor) |

Both inputs are pulled up and fed to the `EXTI9_5_IRQHandler`.

### 5.7 Audio Jack Detect (Optional)

Four pins are configured as inputs with pull-ups to detect cable insertion:

| Signal | GPIO | Pin |
|---|---|---|
| L1 jack detect | GPIOA | PA0 |
| R2 jack detect | GPIOA | PA5 |
| R1 jack detect | GPIOC | PC4 |
| L2 jack detect | GPIOC | PC5 |

### 5.8 SPI Pins (Disabled — SD Card Handled by AVR)

The STM32F4 SPI pins overlapping with the SD card bus are explicitly configured as floating inputs during `initSpiPins()` to avoid bus contention, since the SD card is owned by the ATmega644:

| Signal | GPIO | Pin |
|---|---|---|
| MISO | GPIOA | PA6 |
| MOSI | GPIOA | PA7 |
| SCK | GPIOB | PB3 |
| CS | GPIOB | PB0 |

### 5.9 USB OTG FS

Standard USB OTG full-speed pins (PA11=DM, PA12=DP, PA9=VBUS, PA10=ID) handled by the STM32 USB OTG driver. USB is used to implement **USB MIDI** (class-compliant, no driver required on host).

## 6. Hardware Peripheral Usage

### 6.1 I2S / SPI3 — DAC1 Audio Output (CS4344)

| Parameter | Value |
|---|---|
| Peripheral | SPI3 (I2S mode) on APB1 |
| Standard | I2S Philips |
| Sample rate | 44.1 kHz requested; actual **44002.76 Hz** |
| Data format | 16-bit |
| Mode | Master transmit |
| MCK output | Enabled (256× oversampling clock to codec) |
| DMA | DMA1 Stream7, Channel 0, memory-to-peripheral, halfword width |
| DMA interrupt | `DMA1_Stream7_IRQn`, preemption priority 0, sub-priority 0 |
| Buffer address | `dma_buffer[(dmaPtr & 1) * OUTPUT_DMA_SIZE * 2]` |
| Buffer size | `OUTPUT_DMA_SIZE * 2 = 64` halfwords per transfer |

The DMA operates in **Normal (non-circular) mode** with manual re-arming in the transfer-complete ISR, implementing ping-pong double buffering. The ISR toggles `dmaPtr`, reconfigures the next buffer half address, re-enables the DMA stream, and signals `bCurrentSampleValid` to the main loop to trigger DSP calculation for the next block.

### 6.2 I2S / SPI2 — DAC2 Audio Output (CS4344) or ADC Input (CS5343)

Identical configuration to DAC1 but on SPI2:

| Parameter | DAC2 mode | ADC mode |
|---|---|---|
| DMA stream | DMA1 Stream4, Ch0 | DMA1 Stream3, Ch0 |
| DMA direction | Memory→Peripheral | Peripheral→Memory |
| IRQ | `DMA1_Stream4_IRQn` | `DMA1_Stream3_IRQn` |
| I2S mode | Master TX | Master RX |

In the default `USE_DAC2=1` build, both SPI2 and SPI3 run as stereo DAC outputs, providing four independent audio channels.

### 6.3 USART2 — MIDI DIN

| Parameter | Value |
|---|---|
| Baud rate | **31250** (standard MIDI) |
| Format | 8N1 |
| Mode | Full-duplex (TX and RX enabled) |
| Interrupts | `USART2_IRQHandler` — RX-not-empty, TX-empty |
| TX buffer | `Fifo fifo_midiTx` |
| RX buffer | `Fifo fifo_midiRx` |

Incoming MIDI bytes are buffered via interrupt and drained in the main loop by `uart_processMidi()`, which passes bytes to the MIDI parser. Outgoing MIDI (clock, note data) is queued in the TX FIFO and transmitted by the interrupt handler.

### 6.4 USART3 — Front Panel Link (to ATmega644)

| Parameter | Value |
|---|---|
| Baud rate | **500,000** |
| Format | 8N1 |
| Mode | Full-duplex |
| Interrupts | `USART3_IRQHandler` — RX-not-empty, TX-empty |
| TX buffer | `Fifo fifo_frontTx` |
| RX buffer | `FifoBig fifo_frontRx` (larger capacity to handle preset dumps) |

The STM32F4 sends parameter changes and voice state to the AVR, which displays them on the LCD. The AVR sends encoder/button events and SD card preset data to the STM32F4. The `FifoBig` type is used for the RX buffer because bulk preset data arriving from the AVR can be larger than a standard FIFO.

### 6.5 TIM2 — External Clock Measurement

| Parameter | Value |
|---|---|
| Peripheral | TIM2 (32-bit general-purpose, APB1) |
| Clock source | APB1 timer clock = 84 MHz |
| Prescaler register | 1 (÷2 effective → 42 MHz count rate) |
| Period | 0xFFFFFFFF (free-running, no overflow interrupt) |
| Mode | Upcounting |
| Purpose | Measures time between incoming clock pulses on PC9 (EXTI Line9) to compute the external BPM |

On each falling edge of the clock input (`EXTI9_5_IRQHandler`), the timer counter value is read and fed through a median filter to estimate the pulse period. `COUNTER_DURATION = 1/42000000` converts counts to seconds. The counter is reset to zero after each measurement to give a relative period.

### 6.6 SysTick — System Timebase

| Parameter | Value |
|---|---|
| Reload value | `HCLK / 4000 = 168,000,000 / 4000 = 42,000` |
| Tick period | **250 μs (0.25 ms)** |
| ISR | `SysTick_Handler` increments `volatile uint32_t systick_ticks` |

The `systick_ticks` counter is used throughout the firmware for:
- Sequencer step timing (`seq_tick()` compares `systick_ticks` against `seq_lastTick + seq_deltaT`)
- Trigger pulse duration (`PULSE_LENGTH = PULSE_LENGTH_MS * 4 = 80` ticks = 20 ms)
- MIDI clock sync averaging (`clockSync.c`)

### 6.7 USB OTG FS — USB MIDI

The USB OTG full-speed peripheral implements a class-compliant **USB MIDI** interface. Incoming USB MIDI messages are received by the OTG ISR and buffered; the main loop calls `usb_getMidi(&msg)` to dequeue them. Outgoing MIDI is sent via `usb_sendMidi()`. The USB device descriptor identifies the LXR as a USB MIDI device (no custom driver required on Windows/Mac/Linux). Interrupt handled by `OTG_FS_IRQHandler`.

## 7. Audio Codec Hardware (CS4344 / CS5343)

The **Cirrus Logic CS4344** is a 24-bit stereo DAC driven purely via I2S with no control interface (no SPI/I2C configuration required). The codec's internal sample rate is set entirely by the I2S clock and MCK provided by the STM32F4.

The optional **CS5343** ADC (used when `USE_DAC2=0`) is similarly a slave I2S receiver requiring no control interface.

Both codecs operate at 3.3 V and interface directly to the STM32F4 GPIO. Audio quality in practice is limited to 16-bit by the DMA buffer data format.
